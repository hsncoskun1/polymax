# POLYMAX — Documentation Rules

**Area:** docs/governance · **Authority:** Operator
**Purpose:** Where each type of document goes. What stays in README. What never gets committed.

---

## Permanent document areas

| Area | What goes here |
|------|---------------|
| `README.md` | Project summary, quick start, folder structure, test entry points, links to docs |
| `docs/governance/` | Architectural decisions, branch/PR rules, source-of-truth map, doc placement rules |
| `docs/releases/` | Versioned delivery reports (one per milestone, permanent) |
| `docs/testing/` | Test contracts, regression matrix, tier definitions, Gamma fixture workflow |
| `tools/` | Standalone CLI helpers and maintenance scripts |

---

## Application areas (not documentation)

These areas are governed by their own internal structure. Do not add governance
or documentation files inside them.

| Area | What goes here |
|------|---------------|
| `backend/` | Python source, tests, fixtures |
| `frontend/` | TypeScript/React source, component tests |
| `launcher/` | Application entry point only |
| `config/` | Runtime configuration files |

---

## README scope

**README.md contains:**
- One-paragraph project description
- Quick start commands (launcher, individual services, test commands)
- Folder structure overview
- Links to `docs/` subsections

**README.md does NOT contain:**
- Versioned milestone history or status list
- Governance rules or process documentation
- Delivery report details
- Decision rationale
- Scratch notes or analysis

**Rule:** If a section in README requires more than 5–6 lines to explain a concept,
it belongs in `docs/` — not in README.

---

## Delivery reports

One file per versioned milestone in `docs/releases/`.

**Naming:** `v{major}.{minor}.{patch}-delivery-report.md`

**Required sections:**
1. Scope (what was approved for this step)
2. Changes made (files modified or created)
3. Test results (tier, count, pass/fail)
4. Remaining risks

**Rules:**
- Written after the milestone is complete and tests pass
- Not edited after the fact
- Do not include speculative content or details that cannot be verified from the repo

---

## Governance documents

In `docs/governance/`. These are living documents — updated as decisions are made
or rules change.

**Files:**
- `decision-log.md` — Dated entries, newest first, one paragraph per decision
- `branch-and-pr-rules.md` — Branch naming, scope isolation, commit discipline
- `source-of-truth-map.md` — Authoritative source for every class of information
- `documentation-rules.md` — This file

**Rule:** Governance documents are updated at the milestone where the decision
or rule is introduced. Retroactive edits are only allowed to correct factual errors.

---

## Testing documentation

In `docs/testing/`. These documents describe testing contracts and workflows.

**Rule:** Testing docs cover test design and test operation only. They do not
contain governance decisions, delivery records, or project-level decisions.
If a testing doc needs to reference a governance decision, it links to
`docs/governance/decision-log.md`.

---

## test-results/ scope

`test-results/` contains test execution output files — raw pytest output or
milestone test reports generated by the `test-and-report` workflow.

**Goes in `test-results/`:**
- Pytest session output (pass/fail counts, timing)
- Per-milestone test snapshots

**Does NOT go in `test-results/`:**
- Delivery reports → `docs/releases/`
- Governance decisions → `docs/governance/`
- Architecture analysis → `docs/governance/`
- Temporary notes → not committed at all

---

## What never gets committed

- Temporary analysis notes
- Scratch files, drafts, or "thinking out loud" documents
- Files named `tmp-*`, `draft-*`, `notes-*`, or similar
- Root-level `.md` files other than `README.md` and `CLAUDE.md`

**Rule:** If a file does not have a clear permanent area from the table above,
it should not be committed. Ask the operator before creating a new top-level
directory or a new `docs/` subdirectory.
